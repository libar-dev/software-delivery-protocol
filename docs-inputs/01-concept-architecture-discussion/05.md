Yes. This is the right refinement.

The previous “requirements/unimplemented specs” framing is too requirements-management-ish. I would replace it with a more natural primitive:

> **A spec is a persistent statement of desired system truth.**
>
> It can be tiny and vague at first, then progressively enriched with examples, rules, constraints, design decisions, implementation bindings, and executable verification.

So the direction becomes:

# Use one progressive **Spec** format for ideas, requirements, design specs, implementation specs, and executable tests

Do not model this as:

```txt
idea -> requirement -> design -> implementation -> test
```

Model it as:

```txt
spec
  starts sparse
  gains facets
  gets refined into child specs
  binds to implementation
  becomes executable
  accumulates evidence
```

That aligns with the original goal from the uploaded proposal: the whole lifecycle should live in the same graph/shape, with progressive detail, traceability, validation, and generated views.

The key is that **maturity changes the required completeness, not the object format**.

# The primitive: `Spec`

Instead of `Requirement`, use something like:

```ts
type Spec = {
  id: SpecId;
  title: string;

  kind:
    | "capability"
    | "behavior"
    | "rule"
    | "constraint"
    | "quality"
    | "interface"
    | "workflow"
    | "decision"
    | "example";

  abstraction:
    | "initiative"
    | "domain"
    | "capability"
    | "feature"
    | "rule"
    | "scenario"
    | "operation"
    | "component"
    | "contract";

  readiness:
    | "sketch"
    | "framed"
    | "specified"
    | "designed"
    | "bound"
    | "executable"
    | "verified";

  intent?: IntentFacet;
  behavior?: BehaviorFacet;
  constraints?: ConstraintFacet[];
  model?: DomainModelFacet;
  design?: DesignFacet;
  runtime?: RuntimeFacet;
  bindings?: BindingFacet;
  verification?: VerificationFacet;
  evidence?: EvidenceFacet;

  relations?: SpecRelation[];
  tags?: string[];
};
```

The **same object shape** works for:

```txt
early idea
semi-refined capability
business rule
acceptance example
NFR
API contract
Effect Layer binding
Fastify route binding
test/executable specification
verified delivery evidence
```

The trick is to use **facets**. A spec does not need all facets on day one.

# The better mental model

Do not ask:

```txt
Is this a requirement?
Is this a test?
Is this a design spec?
```

Ask:

```txt
What truth about the system does this spec express?
How abstract or concrete is it?
How ready is it?
What facets does it currently have?
What other specs does it refine, constrain, satisfy, or verify?
```

That makes early ideation feel natural.

Example:

```ts
export const CreateOrder = spec({
  id: "spec:orders.create-order",
  title: "Customer creates an order",

  kind: "behavior",
  abstraction: "feature",
  readiness: "sketch",

  intent: {
    actor: "customer",
    outcome: "turn a valid cart into an order",
    value: "customers can complete purchases"
  },

  relations: [
    refines("spec:checkout.complete-purchase"),
    belongsTo("capability:order-management")
  ]
});
```

This is not an “unimplemented requirement.” It is just a **sparse spec**.

Later, the same spec becomes richer.


Still the same format. More detail.

# Use “examples” as the bridge from idea to executable spec

This is where BDD/Gherkin is useful, but I would not make raw `.feature` files the canonical format.

Gherkin already has a useful hierarchy: `Feature`, `Rule`, and `Example`/`Scenario`; Cucumber describes Gherkin as a structure for executable specifications, and says examples are both specification/documentation and tests. ([cucumber.io][1])

So use the **Gherkin idea**, but store it as typed TypeScript data.

```ts
example({
  id: "spec:orders.create-order.valid-cart",
  title: "Valid cart creates an order",

  given: [
    "a customer has a valid cart",
    "all cart items are available"
  ],

  when: [
    "the customer submits the order"
  ],

  then: [
    "an order is created",
    "an OrderCreated event is emitted"
  ]
});
```

At early maturity, this can be semi-structured prose.

At executable maturity, the same example gets bindings:

```ts
example({
  id: "spec:orders.create-order.valid-cart",
  title: "Valid cart creates an order",

  given: [
    step("customerHasValidCart", { customer: "customer-1" }),
    step("allCartItemsAreAvailable")
  ],

  when: [
    step("submitOrder")
  ],

  then: [
    step("orderIsCreated"),
    step("eventIsEmitted", { event: "OrderCreated" })
  ],

  verification: {
    mode: "executable",
    runner: "vitest",
    test: "test:orders.create-order.valid-cart"
  }
});
```

Same spec. Same example. It just becomes executable.

That gives you a smooth path:

```txt
rough example
  -> clarified example
  -> domain example
  -> executable example
  -> verified example
```

No artificial “requirement object” versus “test object” split.

# The format should have two independent axes

One maturity field is not enough. Use two axes:

```txt
abstraction = what level of thinking is this?
readiness   = how complete / reliable / executable is it?
```

Example:

```txt
abstraction:
  initiative
  capability
  feature
  rule
  scenario
  operation
  component
  contract

readiness:
  sketch
  framed
  specified
  designed
  bound
  executable
  verified
```

This avoids weird cases like:

```txt
high-level capability is "implemented"
low-level test is "draft"
NFR is "specified" but not "executable"
API contract is "bound" but not "verified"
```

Those are all valid states.

A high-level spec can be:

```ts
abstraction: "capability"
readiness: "framed"
```

A concrete acceptance example can be:

```ts
abstraction: "scenario"
readiness: "executable"
```

A route contract can be:

```ts
abstraction: "contract"
readiness: "bound"
```

They all share the same `Spec` shape.

# The practical shape

I would structure each spec with these facets:

```ts
type Spec = {
  id: SpecId;
  title: string;
  kind: SpecKind;
  abstraction: SpecAbstraction;
  readiness: SpecReadiness;

  intent?: {
    actor?: string;
    problem?: string;
    outcome?: string;
    value?: string;
    risks?: string[];
    assumptions?: string[];
    openQuestions?: string[];
  };

  behavior?: {
    examples?: ExampleSpec[];
    rules?: RuleSpec[];
    flows?: FlowSpec[];
  };

  constraints?: {
    kind: "business" | "quality" | "security" | "performance" | "compliance";
    statement: string;
    target?: string;
    measurableBy?: string[];
  }[];

  model?: {
    terms?: string[];
    entities?: string[];
    commands?: string[];
    events?: string[];
    states?: string[];
  };

  design?: {
    components?: string[];
    ports?: string[];
    dependencies?: string[];
    decisions?: string[];
    tradeoffs?: string[];
  };

  runtime?: {
    fastifyRoutes?: string[];
    effectLayers?: string[];
    awilixRegistrations?: string[];
    externalSystems?: string[];
  };

  bindings?: {
    code?: string[];
    tests?: string[];
    schemas?: string[];
    docs?: string[];
  };

  verification?: {
    mode?: "manual" | "reviewed" | "contract" | "executable";
    examples?: string[];
    tests?: string[];
    lastResult?: "unknown" | "passed" | "failed";
  };

  evidence?: {
    implementedBy?: string[];
    verifiedBy?: string[];
    observedIn?: string[];
    lastVerifiedAt?: string;
  };

  relations?: SpecRelation[];
};
```

This looks custom, but the underlying concepts are standard enough:

```txt
intent
behavior
rules
examples
constraints
domain model
design
runtime bindings
verification
evidence
relationships
```

That is much less strange than forcing everything into `Requirement`, `UnimplementedRequirement`, `ImplementedRequirement`, etc.

# How this connects to Fastify + Effect Layers

This model works especially well with the Fastify + Effect direction.

Fastify routes are runtime bindings:

```ts
routeSpec({
  id: "api:POST:/orders",
  satisfies: ["spec:orders.create-order"],
  method: "POST",
  path: "/orders",
  request: "schema:CreateOrderRequest",
  response: "schema:CreateOrderResponse",
  handler: "impl:CreateOrderHandler"
});
```

Effect Layers are runtime/design bindings:

```ts
layerSpec({
  id: "layer:CreateOrderUseCaseLive",
  provides: "impl:CreateOrderUseCase",
  satisfies: ["spec:orders.create-order"],
  requires: [
    "port:OrderRepository",
    "port:EventBus"
  ]
});
```

Effect’s own model is useful here because Layers are explicitly about representing and constructing a program dependency graph; the docs describe layers as constructors for services that manage dependencies during construction, with tags, contexts, services, and layers forming the dependency model. ([Effect][2])

So the canonical graph can connect:

```txt
spec:orders.create-order
  satisfiedBy api:POST:/orders
  satisfiedBy layer:CreateOrderUseCaseLive
  implementedBy impl:CreateOrderUseCase
  verifiedBy spec:orders.create-order.valid-cart

layer:CreateOrderUseCaseLive
  provides impl:CreateOrderUseCase
  requires port:OrderRepository
  requires port:EventBus

api:POST:/orders
  invokes impl:CreateOrderUseCase
```

The spec remains the durable delivery object. Fastify and Effect are implementation/runtime facets.

# How to support large coherent ideation before drilling down

This is where the model becomes powerful.

Add a `specGroup` or `specPack`.

```ts
export const CheckoutV1 = specPack({
  id: "pack:checkout-v1",
  title: "Checkout v1",

  intent: {
    outcome: "allow customers to complete purchases reliably",
    value: "increase conversion and reduce failed orders"
  },

  sharedModel: {
    terms: [
      "Cart",
      "Order",
      "Payment",
      "InventoryReservation",
      "OrderCreated"
    ],
    actors: [
      "Customer",
      "PaymentProvider",
      "Warehouse"
    ]
  },

  constraints: [
    quality({
      kind: "performance",
      statement: "checkout should feel immediate",
      target: "p95 < 500ms for submit order"
    }),
    quality({
      kind: "reliability",
      statement: "payment and order creation must not produce duplicate orders"
    })
  ],

  specs: [
    ref("spec:cart.validate"),
    ref("spec:payment.authorize"),
    ref("spec:orders.create-order"),
    ref("spec:inventory.reserve"),
    ref("spec:notifications.send-order-confirmation")
  ]
});
```

This lets you ideate at the group level:

```txt
What capabilities are missing?
Which specs overlap?
Which domain terms are inconsistent?
Which specs depend on undefined concepts?
Which high-level outcome lacks scenario examples?
Which NFRs affect multiple specs?
Which decisions are needed before drilling down?
Which specs are risky and need examples first?
```

That is exactly the “large coherent group of specs with low detail” workflow.

At this stage, validators should not demand implementation. They should demand **coherence**.

Example sketch-level validators:

```txt
Every spec must have:
- stable id
- title
- kind
- abstraction
- readiness
- intent or parent relation

Every spec pack must have:
- outcome
- shared model or glossary
- specs
- known open questions
- at least one business constraint or success criterion

No spec may:
- use unknown domain terms
- duplicate another spec’s intent without a relation
- depend on a non-existent spec
- contradict a group-level constraint
```

Later, when specs become `designed`, `bound`, or `executable`, stricter validators apply.

# Use validation profiles, not different formats

This is the core mechanism.

```ts
const validationProfiles = {
  sketch: [
    requires("id"),
    requires("title"),
    requires("kind"),
    requiresOneOf("intent.outcome", "relations.parent")
  ],

  framed: [
    requires("intent.outcome"),
    requires("relations"),
    requiresOneOf("behavior.rules", "behavior.examples", "constraints")
  ],

  specified: [
    requiresOneOf("behavior.examples", "constraints"),
    noOpenQuestionsMarkedBlocking()
  ],

  designed: [
    requires("design.components"),
    requires("design.decisions"),
    requiresResolvableRelations()
  ],

  bound: [
    requiresOneOf("runtime.fastifyRoutes", "runtime.effectLayers"),
    requires("bindings.code"),
    requiresResolvableBindings()
  ],

  executable: [
    requires("verification.mode", "executable"),
    requires("verification.tests")
  ],

  verified: [
    requires("evidence.verifiedBy"),
    requiresLastVerificationPassed()
  ]
};
```

Same format. Different expectations.

This is better than:

```txt
IdeaRequirement
DraftRequirement
ImplementedRequirement
VerifiedRequirement
```

because the shape remains stable.

# How to avoid this becoming too custom

Anchor it in familiar, interoperable ideas:

```txt
BDD/Gherkin:
  use Feature / Rule / Example / Given / When / Then semantics for behavior

JSON Schema:
  use declarative schemas for request/response/data constraints

Effect Layers:
  use layer/service dependency graph for runtime construction

Architecture graph:
  use nodes/edges for traceability, projections, validation, and AI context
```

JSON Schema is useful for the contract facet because it is a declarative language for structure and constraints, validators check instances against schemas, and it can support both machine and human-readable documentation. ([json-schema.org][3])

CUE is also worth considering for advanced validation/export because it is designed for data validation with schema and policy constraints coexisting, and its docs explicitly mention validation, configuration, querying, code generation, and flexible generation pipelines. ([CUE][4])

But I would not make CUE or JSON Schema the authoring language for everything. I would use TypeScript as the repo-native authoring DSL and generate/export to those formats where useful.

```ts
import { describe, it, expect } from "vitest";
import { Effect } from "effect";
import { specTest } from "@akg/spec-test";
import { CreateOrderUseCaseTest } from "../../src/orders/create-order.layer";

specTest("test:orders.create-order.valid-cart", {
  verifies: "spec:orders.create-order.valid-cart",

  run: () =>
    Effect.gen(function* () {
      const createOrder = yield* CreateOrderUseCase;

      const result = yield* createOrder.execute(validCartCommand());

      expect(result.orderId).toBeDefined();
      yield* expectEventEmitted("OrderCreated");
    }).pipe(Effect.provide(CreateOrderUseCaseTest))
});
```

The executable test is now just another projection/binding of the same scenario spec.

# The graph changes too

Instead of:

```txt
Requirement -> Implementation -> Test
```

Use:

```txt
Spec -> Spec -> Spec -> Implementation -> Evidence
```

Example:

```txt
pack:checkout-v1
  contains spec:orders.create-order

spec:orders.create-order
  refines spec:checkout.complete-purchase
  dependsOn spec:payment.authorize
  hasExample spec:orders.create-order.valid-cart
  constrainedBy spec:orders.create-order.performance
  designedBy adr:007-order-lifecycle
  satisfiedBy layer:CreateOrderUseCaseLive
  exposedBy api:POST:/orders

spec:orders.create-order.valid-cart
  exemplifies spec:orders.create-order
  verifies spec:orders.create-order
  executedBy test:orders.create-order.valid-cart

test:orders.create-order.valid-cart
  passedOn build:2026-05-17
```

This is more natural and more powerful.

---

# The main design rule

Use **refinement**, not replacement.

A vague spec should not be converted into a different artifact. It should either be:

```txt
enriched in place
```

or:

```txt
refined into child specs
```

Example:

```txt
spec:checkout.complete-purchase
  refines into:
    spec:cart.validate
    spec:payment.authorize
    spec:orders.create-order
    spec:inventory.reserve
    spec:notifications.send-confirmation
```

The parent remains useful for architecture, AI context, roadmap, and business alignment.

The children become more precise.

Some children become executable.

---

# Final recommended direction

I would update the architecture direction to this:

> Build a **repo-native progressive specification graph** where the single durable artifact is `Spec`. A `Spec` can represent an idea, capability, behavior, rule, quality constraint, interface contract, design binding, or executable example using the same shape. Specs start sparse, gain facets over time, refine into child specs, bind to Fastify routes and Effect Layers, and eventually become executable/verified through test bindings. Validators enforce different completeness rules by readiness level, while all documentation, diagrams, traceability matrices, dashboards, and AI slices are generated from the same graph.

This solves the awkwardness.

You no longer have “unimplemented requirements.”

You have:

```txt
sparse specs
framed specs
specified specs
designed specs
bound specs
executable specs
verified specs
```

All in one format.
