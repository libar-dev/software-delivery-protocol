# Recommended answer

Use **Gherkin as the canonical format for behavioral specs**, and parse it into the graph.

Use:

```txt
Feature   -> high-level spec / feature / capability slice
Rule      -> business rule
Scenario  -> concrete example / executable specification
Given     -> precondition / setup
When      -> stimulus / command / event
Then      -> expected observable outcome
Tags      -> graph metadata and relationships
Description -> richer human-readable intent/design notes
Step definitions -> executable binding to code
Cucumber results -> verification evidence
```

This works because Gherkin already gives structure to executable specifications using `Feature`, `Rule`, `Example`/`Scenario`, `Given`, `When`, `Then`, `Background`, `Scenario Outline`, `Examples`, tags, comments, data tables, and doc strings. Cucumber’s reference says examples/scenarios are both specification/documentation and tests, and that the set of examples is an executable specification of the system. ([cucumber.io][1])

So instead of this custom shape:

```ts
spec({
  id: "spec:orders.create-order",
  readiness: "specified",
  behavior: { ... }
});
```

you can write this:

```gherkin
@spec.orders.create-order
@capability.order-management
@readiness.framed
@actor.customer
Feature: Customer creates an order

  Intent:
  The customer can submit a valid cart and receive an order confirmation.

  Value:
  This allows checkout to complete and creates a durable order record.

  Open questions:
  - Should inventory reservation happen before or after payment authorization?
  - Should OrderCreated be emitted before or after confirmation email scheduling?

  Rule: Only valid carts can become orders

    @example.valid-cart
    @readiness.specified
    Scenario: Valid cart creates an order
      Given a customer has a valid cart
      And all cart items are available
      When the customer submits the order
      Then an order is created
      And an OrderCreated event is emitted
```

That is not a weird proprietary spec language. It is ordinary Gherkin, with graph-aware tags.

# The key idea: annotated Gherkin, not custom Gherkin

Do **not** create a new syntax.

Use official Gherkin mechanisms:

```txt
Tags
Descriptions
Feature
Rule
Scenario / Example
Scenario Outline
Examples tables
Doc Strings
Data Tables
```

Tags are especially useful because Cucumber officially supports them on `Feature`, `Rule`, `Scenario`, `Scenario Outline`, and `Examples`; tags are inherited by child elements; and they can be used for running subsets, hooks, documentation, and linking to external IDs. ([cucumber.io][2])

That gives you a clean annotation vocabulary:

```gherkin
@spec.orders.create-order
@capability.order-management
@readiness.sketch
@readiness.framed
@readiness.specified
@readiness.executable
@component.orders-api
@layer.CreateOrderUseCaseLive
@api.POST.orders
@adr.007-order-lifecycle
@nfr.performance
@risk.duplicate-orders
```

For richer prose, use the free-form Gherkin description under `Feature`, `Rule`, or `Scenario`. Cucumber says descriptions can be placed under Feature/Scenario/Rule elements and can be written as Markdown; descriptions are ignored at runtime but available for reporting. ([cucumber.io][1])

That means early ideation can stay natural:

```gherkin
@spec.checkout.complete-purchase
@readiness.sketch
Feature: Complete purchase

  Intent:
  Customers should be able to complete checkout reliably.

  Current thinking:
  Checkout probably needs cart validation, payment authorization,
  order creation, inventory reservation, and customer notification.

  Open questions:
  - Is inventory reservation synchronous?
  - Can payment be authorized before order creation?
  - What happens if notification fails?
```

Later, the same file is enriched:

```gherkin
@spec.checkout.complete-purchase
@readiness.framed
Feature: Complete purchase

  Rule: Customer must have a valid cart

    @spec.cart.validate
    @readiness.specified
    Scenario: Cart contains unavailable item
      Given a customer has a cart with an unavailable item
      When the customer starts checkout
      Then checkout cannot continue
      And the customer sees which item is unavailable

  Rule: Valid checkout creates an order

    @spec.orders.create-order
    @readiness.executable
    @api.POST.orders
    @layer.CreateOrderUseCaseLive
    Scenario: Valid cart creates an order
      Given a customer has a valid cart
      And payment is authorized
      When the customer submits the order
      Then an order is created
      And an OrderCreated event is emitted
```

Same format. More detail. No artifact migration.

# How it feeds the graph

Your extractor would parse Gherkin into graph nodes and edges.

```txt
Feature: Customer creates an order
  -> node: spec.orders.create-order
  -> kind: behavior / feature
  -> readiness: framed
  -> capability: order-management

Rule: Only valid carts can become orders
  -> node: rule.orders.valid-cart-only
  -> edge: constrains spec.orders.create-order

Scenario: Valid cart creates an order
  -> node: example.orders.create-order.valid-cart
  -> edge: exemplifies spec.orders.create-order
  -> readiness: specified or executable

@api.POST.orders
  -> edge: exposedBy api.POST.orders

@layer.CreateOrderUseCaseLive
  -> edge: satisfiedBy layer.CreateOrderUseCaseLive

@adr.007-order-lifecycle
  -> edge: decidedBy adr.007-order-lifecycle
```

Then when Cucumber executes the scenario, execution results become evidence:

```txt
example.orders.create-order.valid-cart
  executedBy test-run:cucumber:2026-05-17
  result: passed
  verifies spec.orders.create-order
```

Cucumber Messages are useful here because they provide a message protocol for Cucumber results and related information; the project states that the protocol decouples Gherkin from Cucumber execution, includes JSON Schema definitions, uses NDJSON streams, and can include Gherkin source, document AST, step definitions, and parameter types. ([GitHub][3])

A practical pipeline would be:

```bash
cucumber-js \
  --format message:generated/cucumber.ndjson

akg ingest-gherkin features/**/*.feature
akg ingest-cucumber-results generated/cucumber.ndjson
akg ingest-code src/**/*.ts
akg validate
akg views
```

# What the graph mapping should look like

A simple mapping table:

| Gherkin construct      | Graph meaning                                                 |
| ---------------------- | ------------------------------------------------------------- |
| `Feature`              | High-level spec, feature, capability slice, or workflow       |
| `Rule`                 | Business rule or invariant                                    |
| `Scenario` / `Example` | Concrete example, acceptance criterion, executable spec       |
| `Scenario Outline`     | Parameterized example family                                  |
| `Examples` table       | Concrete cases / test data                                    |
| `Given`                | Preconditions, initial state, fixtures                        |
| `When`                 | Command, event, user action, system stimulus                  |
| `Then`                 | Observable expected outcome                                   |
| Tags                   | IDs, relationships, maturity/readiness, architecture bindings |
| Description            | Intent, value, assumptions, open questions, design notes      |
| Step definition        | Executable binding to code                                    |
| Test result            | Verification evidence                                         |

This gives you the progressive format you wanted:

```txt
Idea
  = Feature with intent and open questions

Semi-refined spec
  = Feature + Rules + rough Scenarios

Implementation-ready spec
  = Feature + Rules + precise Scenarios + examples + architecture tags

Executable spec
  = Same Scenario with step definitions

Verified spec
  = Same Scenario with passing result evidence
```

## 1. Sketch

```gherkin
@spec.orders.create-order
@readiness.sketch
@capability.order-management
Feature: Customer creates an order

  Intent:
  A customer should be able to create an order from a valid cart.

  Open questions:
  - What happens if payment succeeds but order persistence fails?
  - Should inventory be reserved before or after order creation?
```

## 2. Framed

```gherkin
@spec.orders.create-order
@readiness.framed
@capability.order-management

  Rule: Only valid carts can become orders

  Rule: Creating an order emits OrderCreated

  Rule: Duplicate submissions must not create duplicate orders
```

## 3. Specified

```gherkin
@spec.orders.create-order
@readiness.specified
@capability.order-management
Feature: Customer creates an order

  Rule: Only valid carts can become orders

    @example.valid-cart
    Scenario: Valid cart creates order
      Given a customer has a valid cart
      And all items are available
      When the customer submits the order
      Then an order is created
      And an OrderCreated event is emitted

    @example.unavailable-item
    Scenario: Cart contains unavailable item
      Given a customer has a cart with an unavailable item
      When the customer submits the order
      Then no order is created
      And the customer is told which item is unavailable
```

## 4. Designed

```gherkin
@spec.orders.create-order
@readiness.designed
@capability.order-management
@component.orders-api
@component.orders-domain
@adr.007-order-lifecycle
Feature: Customer creates an order

  Rule: Only valid carts can become orders

    @example.valid-cart
    @api.POST.orders
    @layer.CreateOrderUseCaseLive
    Scenario: Valid cart creates order
      Given a customer has a valid cart
      And all items are available
      When the customer submits the order
      Then an order is created
      And an OrderCreated event is emitted
```

## 5. Executable

Same Gherkin, now with TypeScript step definitions:

```ts
import { Given, When, Then } from "@cucumber/cucumber";
import { expect } from "vitest";
import { Effect } from "effect";
import { testRuntime } from "../support/test-runtime";
import { CreateOrderUseCase } from "../../src/orders/create-order.layer";

Given("a customer has a valid cart", async function () {
  this.cart = await fixtures.createValidCart();
});

Given("all items are available", async function () {
  await fixtures.makeAllItemsAvailable(this.cart);
});

When("the customer submits the order", async function () {
  this.result = await Effect.runPromise(
    Effect.gen(function* () {
      const createOrder = yield* CreateOrderUseCase;
      return yield* createOrder.execute({ cartId: this.cart.id });
    }).pipe(Effect.provide(testRuntime))
  );
});

Then("an order is created", function () {
  expect(this.result.orderId).toBeDefined();
});

Then("an OrderCreated event is emitted", async function () {
  await expectEvent("OrderCreated", { orderId: this.result.orderId });
});
```

Now the scenario is not just prose. It is executable evidence.

# Where Gherkin is a great fit

Use Gherkin for:

```txt
business behavior
acceptance criteria
cross-service workflows
user journeys
domain rules
observable outcomes
integration tests
API behavior from a user/system perspective
executable examples
large-scale coherent ideation
```

Cucumber’s own guidance says `Given` describes initial context, `When` describes an event/action, and `Then` describes an expected observable outcome; it also encourages hiding implementation details in step definitions. ([cucumber.io][1])

That is exactly what you want for high-quality specs before drilling into implementation.

---

# Where Gherkin is not enough

Do **not** force everything into Gherkin.

Gherkin is weak for:

```txt
low-level unit tests
architecture dependency rules
type-level contracts
data schema constraints
internal algorithm properties
Effect Layer composition checks
Fastify plugin registration checks
component dependency policies
```

For those, use normal TypeScript tests and annotate them.

Example with a thin Vitest-style wrapper:

```ts
import { specTest } from "@akg/test";
import { describe, expect } from "vitest";

describe("Order lifecycle architecture", () => {
  specTest("arch.orders.no-api-to-db", {
    verifies: ["spec.orders.create-order"],
    kind: "architecture-rule",
    readiness: "executable"
  }, async graph => {
    expect(
      graph.component("component.orders-api")
        .hasDirectDependencyOn("external.postgres")
    ).toBe(false);
  });
});
```

For Playwright E2E/UI tests, use its built-in tags and annotations. Playwright Test supports tagging tests with `@fast`, `@slow`, etc., filtering by tag, and richer annotations with a `type` and `description` exposed to reporters. ([Playwright][4])

Example:

```ts
import { test, expect } from "@playwright/test";

test("customer can submit valid order", {
  tag: [
    "@spec.orders.create-order",
    "@readiness.executable",
    "@capability.order-management"
  ],
  annotation: [
    {
      type: "spec",
      description: "spec.orders.create-order"
    },
    {
      type: "api",
      description: "api.POST.orders"
    }
  ]
}, async ({ page }) => {
  // normal Playwright test
});
```

So the model becomes:

```txt
Gherkin
  for business-readable behavior specs

Cucumber step definitions
  for executable acceptance/integration specs

Vitest/Jest-style helpers
  for unit, architecture, and technical executable specs

Playwright annotations/tags
  for browser/user journey specs

Fastify/Effect annotations
  for runtime implementation bindings

Architecture graph
  as the normalized canonical model
```

# One important caveat

Gherkin tags are not type-safe by themselves.

This:

```gherkin
@spec.orders.creat-order
```

can be misspelled.

So you need an `akg lint` step:

```bash
akg lint features/**/*.feature
```

It should check all tags against generated registries and extracted code:

```txt
Unknown spec id:
  @spec.orders.creat-order

Did you mean:
  @spec.orders.create-order?
```

You can also generate TypeScript unions from parsed Gherkin:

```ts
export type SpecId =
  | "spec.orders.create-order"
  | "spec.checkout.complete-purchase";
```

Then code annotations and test wrappers can use type-safe IDs:

```ts
specTest("orders create order", {
  verifies: "spec.orders.create-order" satisfies SpecId
}, async () => {
  // ...
});
```

That gives you the best compromise:

```txt
Developers write normal Gherkin and normal tests.
The graph tooling validates and normalizes everything.
TypeScript gets generated types for code-side references.
CI enforces consistency.
```

# Final direction

Yes: **annotated Gherkin should be the main human-facing spec format**.

Use it like this:

```txt
Gherkin Feature/Rule/Scenario
  = progressive behavior specification

Gherkin tags
  = graph annotations and relationships

Gherkin descriptions
  = intent, assumptions, open questions, design notes

Cucumber step definitions
  = executable binding

Cucumber Messages/results
  = verification evidence

TypeScript test wrappers / Playwright annotations
  = technical and lower-level executable specs

Architecture graph
  = canonical normalized model behind all of them
```

This avoids a drastic new practice. People can write familiar BDD-style specs and normal TypeScript tests, while your tooling quietly turns them into a typed, queryable, validated delivery graph.
