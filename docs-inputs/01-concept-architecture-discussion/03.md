Yes, it makes sense. The underlying idea is strong: **treat the codebase as a typed, queryable model of the whole software delivery process**, not just as source files. The uploaded write-up already has the right ingredients: annotation-based marking, `ts-morph` extraction, graph construction, traceability, validation, and generated views/docs.

The direction I would converge on is this:

# Build a repo-native **Architecture Knowledge Graph as Code** for TypeScript

Not a language-server tool. Not a documentation generator. Not a giant external requirements system.

A practical version is:

> **Typed delivery/architecture model in the repo + lightweight code markers + static extractor + graph validators + generated human/AI views.**

The key design decision is: **do not make decorators the whole system.** Use decorators or JSDoc-style annotations mainly to bind real code locations to graph concepts. Keep the durable delivery state—requirements, capabilities, decisions, components, maturity, trace links—in explicit, typed TypeScript model files.

That gives you the best of both worlds: code-adjacent truth, type safety, deterministic extraction, and no dependence on fragile runtime decorator metadata.

---

## The practical architecture

Use four layers.

## 1. Typed delivery model: the source of truth

Create a folder like:

```txt
/arch
  requirements.ts
  capabilities.ts
  components.ts
  decisions.ts
  nfrs.ts
  views.ts
```

This is where you define business and architecture state as typed TypeScript.

Example:

```ts
// arch/requirements.ts
import { defineRequirements } from "@akg/model";

export const requirements = defineRequirements({
  "FR-042": {
    kind: "functional",
    title: "Customer can create an order",
    maturity: "designed",
    priority: "P0",
    capability: "capability:order-management",
    acceptanceCriteria: [
      "Customer can submit a valid cart",
      "System persists the order",
      "System emits OrderCreated event"
    ],
    expectedImplementations: [
      "impl:CreateOrderUseCase"
    ],
    expectedTests: [
      "test:CreateOrderUseCase"
    ],
    decisions: [
      "adr:007-order-lifecycle"
    ]
  },

  "NFR-003": {
    kind: "non-functional",
    title: "Order creation latency",
    maturity: "specified",
    qualityAttribute: "performance",
    target: "p95 < 300ms"
  }
} as const);
```

The important part is that **ideas, requirements, designs, implementations, tests, and decisions use the same graph shape**, but different maturity levels require different fields. This matches the uploaded proposal’s “idea → specified → designed → implemented → verified” progression and its typed node/edge approach.

Use TypeScript discriminated unions for maturity:

```ts
type Maturity =
  | "idea"
  | "draft"
  | "specified"
  | "designed"
  | "implemented"
  | "verified";

type Requirement =
  | IdeaRequirement
  | DraftRequirement
  | SpecifiedRequirement
  | DesignedRequirement
  | ImplementedRequirement
  | VerifiedRequirement;
```

Then validators enforce rules like:

```ts
implemented requirement must have:
- at least one implementation
- at least one test
- owner
- linked capability
- no broken references
```

This avoids the trap of “everything is prose.” The delivery process becomes typed, versioned, reviewable code.

---

## 2. Lightweight source-code markers

In product code, mark significant abstractions.

Use decorators where they are natural:

```ts
import { arch } from "@akg/markers";

@arch.node({
  id: "impl:CreateOrderUseCase",
  kind: "use-case",
  implements: ["FR-042"],
  component: "component:orders-api",
  capability: "capability:order-management"
})
export class CreateOrderUseCase {
  async execute(command: CreateOrderCommand) {
    // implementation
  }
}
```

For functions, constants, event handlers, tests, or places where decorators are awkward, use JSDoc or exported metadata:

```ts
/**
 * @akg.node impl:OrderCreatedEventHandler
 * @akg.kind event-handler
 * @akg.handles event:OrderCreated
 * @akg.supports FR-042
 */
export async function handleOrderCreated(event: OrderCreated) {
  // implementation
}
```

or:

```ts
export const createOrderUseCaseAkg = markImplementation({
  id: "impl:CreateOrderUseCase",
  implements: ["FR-042"],
  component: "component:orders-api"
});
```

This is important because modern TypeScript decorators are useful, but they have practical constraints. TypeScript 5.0 introduced support for the newer decorator proposal, while also noting differences from legacy experimental decorators, including that the new decorators are not compatible with `--emitDecoratorMetadata` and do not support parameter decorators. ([Microsoft for Developers][1])

So the safer rule is:

> **Use decorators as syntax the static extractor can read, not as runtime reflection infrastructure.**

That gives you annotation-driven marking without coupling the whole system to runtime metadata.

---

## 3. Static extractor builds the graph

Build a CLI, for example:

```bash
pnpm akg build
```

It does this:

```txt
typed model files
        +
decorators / JSDoc / marker constants
        +
inferred code relationships
        ↓
canonical architecture graph
```

Use `ts-morph` as the extraction engine. It wraps the TypeScript compiler API, which is exactly what you need for source navigation and AST manipulation. ([ts-morph][2]) It can retrieve decorators with `getDecorators()`, inspect decorator names, check decorator factory arguments, and even add/remove decorators for codemod-style automation. ([ts-morph][3])

The graph should be simple at first:

```ts
export interface GraphNode {
  id: string;
  kind:
    | "requirement"
    | "nfr"
    | "capability"
    | "component"
    | "use-case"
    | "implementation"
    | "test"
    | "decision"
    | "event"
    | "external-system";

  maturity?: Maturity;
  title?: string;
  location?: {
    file: string;
    line: number;
  };
  metadata?: Record<string, unknown>;
}

export interface GraphEdge {
  from: string;
  to: string;
  type:
    | "refines"
    | "realizes"
    | "implements"
    | "verifies"
    | "dependsOn"
    | "contains"
    | "emits"
    | "handles"
    | "decidedBy"
    | "constrainedBy";

  source: "declared" | "annotation" | "inferred";
  confidence?: "high" | "medium" | "low";
}
```

Do not try to infer everything. Start with:

```txt
Declared edges:
- requirement -> capability
- requirement -> ADR
- requirement -> expected implementation
- implementation -> test

Annotation edges:
- class/function -> requirement
- class/function -> component
- test -> requirement
- event handler -> event

Inferred edges:
- imports
- inheritance
- implements interface
- uses class/function
- test imports implementation
```

The uploaded write-up already points in this direction: explicit edges from annotations, inferred edges from static analysis, traceability between requirements/code/tests/ADRs, and graph serialization into a canonical model.

## 4. Validators make it useful

The graph becomes valuable when it fails CI.

Example commands:

```bash
pnpm akg validate
pnpm akg views
pnpm akg check
```

Validation rules should start very concrete:

```txt
Requirement rules:
- Every implemented requirement must have at least one implementation.
- Every implemented requirement must have at least one verifying test.
- Every requirement referenced from code must exist.
- Every requirement above "draft" must belong to a capability.
- Every NFR above "specified" must have a measurable target.

Architecture rules:
- Every implementation must belong to a component.
- Every component must belong to a capability or bounded context.
- Forbidden dependencies fail the build.
- Cycles between architectural layers fail the build.
- No implementation may reference an ADR that does not exist.

Traceability rules:
- Every test claiming to verify a requirement must point to a real requirement.
- Every ADR referenced by implementation must exist.
- Every public API endpoint must trace to at least one requirement or capability.
```

This is the “compilation guarantees or similar” part. Some constraints will be enforced by TypeScript types; others need custom graph validators. The uploaded document also frames this correctly: TypeScript helps, but custom graph validators, architecture tests, and CI checks are what actually catch missing links, orphan requirements, invalid maturity transitions, and broken references.

# What to generate from the graph

Once the graph exists, generate views. Do not hand-maintain them.

Start with four projections:

## 1. Requirements traceability matrix

```txt
Requirement | Maturity | Capability | Implemented By | Verified By | ADRs | Status
FR-042      | designed | Orders     | CreateOrder... | missing     | ADR-007 | incomplete
```

## 2. Architecture diagrams

Use LikeC4 for C4-style architecture views. LikeC4 is specifically positioned as “Architecture as Code” with a DSL for components, relationships, and boundaries, plus generated diagrams, validation, static site output, and AI-friendly access. ([LikeC4][4]) The C4 model itself is a good fit because it uses hierarchical abstractions—software systems, containers, components, and code—and matching diagrams. ([C4 model][5])

Practical recommendation:

```txt
AKG graph -> generate LikeC4 model/views -> diagrams/static site
```

or:

```txt
LikeC4 model -> imported as high-level graph nodes
```

But I would not make LikeC4 the canonical delivery model. Use it mainly for architecture visualization.

## 3. AI-friendly graph slices

Generate JSON files like:

```txt
/generated/ai/order-management.context.json
/generated/ai/unimplemented-requirements.json
/generated/ai/component-orders-api.json
/generated/ai/change-impact-FR-042.json
```

Each slice should include:

```ts
{
  nodes: [],
  edges: [],
  sourceFiles: [],
  unresolvedQuestions: [],
  validationFindings: []
}
```

This gives AI tools structured context instead of throwing the whole repo into a prompt.

## 4. Delivery status dashboard

Generate:

```txt
- capability maturity map
- requirements by maturity
- requirements without implementation
- implementations without requirements
- tests without traceability
- ADRs referenced by code
- components with forbidden dependencies
```

This turns the graph into delivery management, not just architecture documentation.

# What I would not build first

Avoid these in v1:

```txt
- full custom VS Code extension
- graph database
- custom TypeScript transformer
- runtime reflection system
- automatic semantic understanding of every function
- huge decorator taxonomy
- trying to model the entire enterprise delivery lifecycle immediately
```

A graph database like Neo4j may become useful later, but for v1, a generated `architecture.graph.json` in the repo is enough.

CALM is worth watching or using as an export/import format if you want a more standardized architecture-as-code representation. It is an open-source FINOS specification intended to define, validate, and visualize architectures in a machine-readable format. ([CALM][6]) But I would treat it as interoperability, not the heart of the system.

# Concrete MVP direction

Build this as an internal monorepo package:

```txt
/packages/akg-model
  defineRequirement()
  defineCapability()
  defineComponent()
  defineDecision()
  defineView()

/packages/akg-markers
  @arch.node()
  @arch.implements()
  @arch.verifies()
  JSDoc marker parser types

/packages/akg-cli
  akg build
  akg validate
  akg views
  akg explain

/arch
  requirements.ts
  capabilities.ts
  components.ts
  decisions.ts
  views.ts

/generated
  architecture.graph.json
  requirements.md
  traceability-matrix.md
  likec4/
  ai/
```

## First target

Pick **one bounded context** or feature area.

For example:

```txt
Order Management
```

Model only:

```txt
- 10–30 requirements
- 3–6 capabilities
- 3–10 components
- 3–5 ADRs
- 5–20 implementation markers
- 5–20 test markers
```

Then prove the system can answer:

```txt
What implements FR-042?
What tests verify FR-042?
Which requirements are designed but not implemented?
Which components are involved in Order Management?
Which ADRs influenced this feature?
What code would be impacted if this requirement changes?
Which implementations have no requirement trace?
Which requirements have no tests?
```

If it can answer those reliably, the idea is useful.

# Suggested first implementation sequence

## Step 1 — Define the minimal metamodel

Do this first:

```ts
type NodeKind =
  | "requirement"
  | "nfr"
  | "capability"
  | "component"
  | "implementation"
  | "test"
  | "decision";

type EdgeKind =
  | "refines"
  | "realizes"
  | "implements"
  | "verifies"
  | "dependsOn"
  | "decidedBy"
  | "constrainedBy";
```

Do **not** start with 50 node types.

## Step 2 — Create typed registries

Example:

```ts
export const capabilities = defineCapabilities({
  "capability:order-management": {
    title: "Order Management",
    owner: "domain:commerce",
    maturity: "designed"
  }
} as const);
```

## Step 3 — Add source markers

Example:

```ts
@arch.node({
  id: "impl:CreateOrderUseCase",
  kind: "implementation",
  implements: ["FR-042"],
  component: "component:orders-api"
})
export class CreateOrderUseCase {}
```

## Step 4 — Build `akg build`

Output:

```txt
/generated/architecture.graph.json
```

## Step 5 — Build `akg validate`

Fail on:

```txt
- broken IDs
- missing implementation
- missing tests
- invalid maturity transition
- architecture dependency violations
- duplicate node IDs
```

## Step 6 — Generate three useful views

Start with:

```txt
/generated/requirements.md
/generated/traceability.md
/generated/ai/order-management.json
```

Then add LikeC4 diagrams.

The uploaded proposal’s “starting small” path—prototype a handful of decorators/markers, build a `ts-morph` script that dumps a graph, add visuals, then add validators—is exactly the right MVP sequence.

---

# The refined direction in one sentence

Build a **TypeScript-native Architecture Knowledge Graph as Code** where **typed repo files define delivery intent**, **source annotations bind intent to implementation**, **`ts-morph` extracts and enriches the graph**, **validators enforce traceability and architecture rules in CI**, and **all docs, diagrams, dashboards, and AI context are generated projections**.

That is the practical version of the idea. It keeps the ambition, but makes it implementable.
